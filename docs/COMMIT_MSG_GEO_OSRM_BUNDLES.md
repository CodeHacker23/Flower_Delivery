## Body (полное описание для GitHub)

### Реализовано

**Геокодирование (DaData):**
- Подключён **API стандартизации DaData** (Clean): первым вызывается `cleaner.dadata.ru/api/v1/clean/address`, при неудаче или отсутствии `dadata.secret-key` — fallback на API подсказок (Suggest).
- **Ограничение по городу:** в Suggest передаётся `locations: [{ city: "Челябинск" }]`. Результаты Clean проверяются: если DaData вернул другой город (например Копейск для «Цвиллинга 45») — результат отбрасывается, используется Suggest. Так точки не попадают в соседние города.
- **Проверка точности qc_geo:** при `qc_geo >= 2` (улица/город, а не дом) пишется предупреждение в лог.
- Очистка адреса перед геокодом (подъезд, квартира) без изменений.

**Связки заказов и маршруты:**
- **Порядок по маршруту:** при взятии связки заказы переупорядочиваются по оптимальному маршруту (OSRM Trip при заданном `app.osrm.url`, иначе Haversine). Текст «Связка взята!» и кнопки карт показывают один и тот же порядок: забор1 → доставка1 → забор2 → доставка2.
- **2ГИС:** в ссылку маршрута передаётся **первая точка = гео курьера**, затем заборы и доставки. Курьер открывает карту «от себя» и может сразу нажать «Поехали».
- **Яндекс** уже строился от текущего местоположения (`~` в rtext), без изменений.
- Из сообщения «Связка взята!» убрана длинная подсказка про Яндекс/2ГИС и «не нажимай Оптимизировать».

**Миграция БД V9 и тестовые скрипты:**
- **V9** добавляет колонки `orders.shop_pickup_address_override`, `shop_pickup_latitude`, `shop_pickup_longitude` — **только для тестирования связок**. В проде они должны быть NULL (используется адрес магазина из `shops`).
- Скрипты подстановки тестовых адресов забора: `replace_shop_pickup_for_test.sql`, `replace_shop_pickup_15.sql`, `run_replace_shop_pickup.sql` — перед продакшеном не запускать; данные откатываются скриптом `revert_shop_pickup_override.sql`.
- `fix_delivery_coords_for_test.sql` — правка координат доставки для типичных адресов (например Комсомольский пр-кт 41 в ЧЛБ). По необходимости запускать в тесте; в проде координаты приходят из геокодера при создании заказа.
- `deposit_topup_642867793.sql` — пример пополнения баланса по telegram_id (для теста/удобства).

### Обязательно перед продакшеном (вернуть исходное состояние)

| Действие | Что сделать |
|----------|-------------|
| **Откат override адресов забора** | Выполнить `scripts/revert_shop_pickup_override.sql`: обнулить `shop_pickup_address_override`, `shop_pickup_latitude`, `shop_pickup_longitude` у всех заказов. После этого адрес забора берётся только из `shops.pickup_address`. |
| **Не запускать тестовые подстановки** | Не выполнять на прод-БД: `run_replace_shop_pickup.sql`, `replace_shop_pickup_for_test.sql`, `replace_shop_pickup_15.sql`. Они подменяют адреса забора на тестовые для проверки связок. |
| **Проверка 200 м (гео)** | Раскомментировать блок проверки радиуса 200 м в `CourierGeoHandler.handleLocation()` (TODO в коде). Включить `app.penalty.skip-geo-check=false` в `application.properties`. |
| **Временные команды бота** | По желанию: убрать или ограничить команды активации магазина/курьера для теста (в `Bot.java` помечены как «ВРЕМЕННАЯ КОМАНДА»). |

Колонки V9 в БД **не удалять**: в проде они просто остаются NULL. Удалять миграцию из Flyway не нужно — иначе рассинхрон с уже применёнными окружениями.

### Кратко по гео и связкам

- **Конфиг:** в `application.properties` только город Челябинск (`app.region.city=Челябинск`). Геокодер ограничен этим городом (Suggest — через `locations`, Clean — отбрасываем результат, если вернулся другой город).
- **OSRM:** при указанном `app.osrm.url` используется свой экземпляр OSRM по Челябинску (Trip API для порядка точек в связке, Route — при необходимости). Иначе — расчёт по прямой (Haversine).
- **Связки:** рекомендуемые связки 2–3 заказа считаются по кратчайшему маршруту; после взятия связки порядок в сообщении и в ссылках Яндекс/2ГИС совпадает, первая точка в 2ГИС — геолокация курьера.
