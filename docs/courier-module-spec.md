# Курьерский модуль: спецификация и план реализации

Документ описывает, что уже сделано для курьеров и что **обязательно** нужно реализовать по ТЗ, в том числе по депозитам, комиссиям и защите от обхода. **Геолокация курьера нужна в любом случае** — без неё невозможна проверка «был ли курьер в магазине/у получателя» и система штрафов.

## 2. Геолокация курьера — обязательный блок

**Зачем нужна в любом варианте (в т.ч. базовом MVP):**

- Проверка при отмене/возврате: был ли курьер в магазине и у получателя (в радиусе ~200 м).
- Основа для штрафов: «заказ отменён, но гео показывает выполнение → штраф 500₽».
- Защита от обхода: нельзя «накрутить» статусы без реального присутствия.

### 2.1 Что нужно сделать

1. **Хранение геолокации курьера**
   - Таблица или поля: `courier_id`, `latitude`, `longitude`, `updated_at` (последняя известная точка).
   - Вариант: отдельная таблица `courier_locations` (история) + поле «последняя точка» в `couriers` или в отдельной таблице `courier_last_location` (одна запись на курьера).

2. **Обновление геолокации**
   - **Вариант 1 (базовый MVP):** курьер **вручную** отправляет локацию в боте (кнопка «Поделиться геолокацией» или сообщение с `location`). Бот принимает `Update` с `message.getLocation()`, сохраняет в БД.
   - **Вариант 2 (оптимизированный MVP):** автоматическое обновление раз в 15–30 сек (требует фоновый процесс/периодический запрос к клиенту или отдельное приложение).

3. **Привязка гео к заказу (для проверок)** — **вариант «2 в 1»**
   - Вместо отдельной кнопки «В магазине» и отдельной отправки гео — **подтверждение статуса = отправка геолокации**.
   - **Сценарий:** курьер нажимает inline-кнопку «В магазине» (или «Вручил») → бот **не** меняет статус сразу, а отправляет сообщение: «Отправьте геолокацию, чтобы подтвердить, что вы в магазине» и показывает клавиатуру с кнопкой **«📍 Отправить геолокацию»** (ReplyKeyboard с `request_location: true`). Курьер нажимает эту кнопку и Telegram присылает локацию → бот в одном действии: (1) переводит заказ в статус «В магазине» (или «Вручил»), (2) сохраняет снимок гео (order_id, статус, lat, lon, courier_id, created_at). Одна отправка локации = и подтверждение статуса, и фиксация гео для проверки при отмене/штрафах.
   - То же для «Вручил»: кнопка «Вручил» → запрос геолокации у точки доставки → после получения локации: статус DELIVERED + снимок гео у получателя.
   - Плюсы: курьер не делает лишних шагов, гео всегда привязано к моменту подтверждения статуса, нельзя «подтвердить в магазине» без отправки локации.

4. **Координаты магазина и точки доставки**
   - У магазина уже есть адрес забора — нужно геокодирование адреса забора (как у доставки) и сохранение `shop.pickup_lat`, `shop.pickup_lon` (или уже есть — проверить модель Shop).
   - У заказа/точек доставки уже есть `delivery_latitude`, `delivery_longitude` (геокодирование при создании заказа).

5. **Функция проверки «в радиусе 200 м»**
   - Утилита: `distance(lat1, lon1, lat2, lon2)` (Haversine) и проверка `distance <= 0.2` км (200 м).
   - Использовать при: отмене заказа, возврате, расчёте штрафов.

Итого по гео: **модель/таблица локаций курьера**, **приём локации от курьера в боте**, **сохранение снимков гео при ключевых статусах заказа**, **проверка «в радиусе 200 м»** для магазина и получателя.

**Рекомендация:** для статусов «В магазине» и «Вручил» использовать **подтверждение через геолокацию (2 в 1)**: кнопка статуса не меняет статус сама по себе, а запрашивает локацию; после получения локации — сразу и смена статуса, и сохранение снимка гео. Так не нужно отдельно «подтвердить статус» и «отправить гео».

## 3. Система депозитов курьеров

### 3.1 Таблица депозитов

**Реализовано сейчас:**

- В таблицу `couriers` добавлены поля:
  - `balance DECIMAL(10,2) NOT NULL DEFAULT 0` — текущий баланс депозита курьера;
  - `commission_percent DECIMAL(5,2) NOT NULL DEFAULT 20.00` — процент комиссии с доставки (по умолчанию 20%).
- В модели `Courier` поля `balance` и `commissionPercent` с дефолтами.
- В `CourierService`:
  - `chargeFromBalance(User courierUser, BigDecimal amount)` — аккуратное списание с баланса (false, если денег не хватает);
  - `addToBalance(User courierUser, BigDecimal amount)` — пополнение/возврат комиссии.
- В `OrderService.assignOrderToCourier`:
  - при назначении заказа считается комиссия = `order.getTotalDeliveryPrice() * commission_percent/100`;
  - перед назначением заказа пытаемся списать эту сумму с баланса через `chargeFromBalance`;
  - если денег мало — заказ курьеру **не назначается** (возврат `Optional.empty()` и лог в журнал).

**План на будущее:**

- Отдельная таблица `courier_deposits` не нужна на старте: баланс уже есть в `couriers`.
- При появлении истории операций → отдельная таблица транзакций (см. ниже).

### 3.2 Пополнение депозита

- В боте: кнопка «Пополнить депозит» → выбор суммы (или ввод) → переход в платёжную систему (ЮKassa/Stripe).
- Webhook или callback от платёжки → начисление на баланс курьера + запись в историю транзакций.
- Минимальная сумма пополнения (например, 500₽) — по желанию.

### 3.3 Автоматическое списание комиссии (20% с заказа)

- **Момент списания:** при принятии заказа курьером (assign order) или при переводе заказа в статус «Вручил» (доставлен). По ТЗ логичнее — **при принятии заказа** (комиссия резервируется/списывается сразу).
- Формула: `commission = order.deliveryPrice * (commissionRate / 100)`. Для мультиадресного заказа — от полной суммы доставки заказа (total delivery price).
- **Комиссия настраиваемая:** в настройках системы или в таблице `couriers` поле `commission_percent` (по умолчанию 20). Админка потом выставляет курьеру свой % (на перспективу: больше заказов — меньше %).
- Проверка перед принятием заказа: `courier.balance >= commission`. Если баланса не хватает — «Пополните депозит» и заказ взять нельзя.

**Реализация (черновик, без платёжки):**

- При взятии заказа (`OrderService.assignOrderToCourier`):
  - считаем комиссию через `calculateCommissionForCourier(order, courierUser)`;
  - если комиссия > 0 — пытаемся списать её с баланса через `CourierService.chargeFromBalance`:
    - если денег **не хватает** — заказ не назначаем (Optional.empty), а курьер в хендлере получает понятное сообщение «не хватает денег на депозите для комиссии»;
    - если хватает — обновляем баланс и пишем транзакцию `COMMISSION_CHARGE` в `courier_transactions` через `CourierTransactionService`.
- В «💰 Моя статистика» курьер видит:
  - сколько доставлено заказов и на какую сумму (всего и за месяц),
  - **баланс депозита** и текущий **процент комиссии**.

### 3.4 Проверка баланса депозита

- В боте: пункт меню «Баланс» / «Мой депозит» или блок в «Моя статистика»: текущий баланс, последние операции.
- При попытке взять заказ — проверка достаточности баланса с учётом комиссии.

## 4. Система комиссий (детализация)

- **Расчёт комиссии:** `commission = order.getTotalDeliveryPrice() * (courier.getCommissionPercent() / 100)`, по умолчанию 20%.
- **Списание с депозита:** при назначении заказа на курьера (`assignOrderToCourier`) списывать `commission` с `courier.balance`, записывать транзакцию (см. ниже).
- **Возврат комиссии:** при отмене заказа (до того как курьер «вручил») — вернуть сумму комиссии на баланс курьера и записать обратную транзакцию. При штрафе за обход — не возвращать или списать дополнительно (см. раздел 5).

### 4.1 История транзакций

Таблица, например `courier_transactions`:

- `id`, `courier_id`, `type` (ENUM: DEPOSIT_TOP_UP, COMMISSION_CHARGE, COMMISSION_REFUND, FINE, PENALTY_500, PENALTY_1000 и т.д.), `amount` (положительное — пополнение/возврат, отрицательное — списание), `order_id` (nullable, для операций по заказу), `description` (текст для лога), `created_at`.

По ней: вывод «История операций» в боте и отчётность для админа.

## 5. Защита от обхода (курьерка по ТЗ)

### 5.1 Логика проверки при отмене заказа

Если заказ **отменён** (магазином, админом или курьером), но по данным системы курьер «выполнил» этапы:

- Проверки:
  1. Проставил ли курьер статус «В магазине»? (да/нет)
  2. Проставил ли курьер статус «У получателя» / «Вручил»? (да/нет)
  3. Был ли курьер в радиусе 200 м от магазина (в момент или около момента статуса «В магазине»)? — по сохранённой геолокации.
  4. Был ли курьер в радиусе 200 м от точки доставки (в момент или около момента «Вручил»)? — по сохранённой геолокации.

- Если **все четыре условия «да»** → считаем: заказ фактически выполнен, но отменён → **штраф 500₽** (списание с депозита, запись в транзакции и в историю заказа/курьера).
- Если у курьера **много отменённых заказов** (порог задаётся в настройках) → дополнительно **штраф 1000₽** (разовый или за период — уточнить по ТЗ).

### 5.2 Что нужно в коде

- При смене статусов «В магазине» и «Вручил» сохранять снимок геолокации курьера (последняя полученная точка или за последние N минут) с привязкой к заказу (таблица `order_status_geo_snapshots`: order_id, status, courier_lat, courier_lon, created_at).
- При отмене заказа вызывать сервис проверки: сравнить эти снимки с координатами магазина и доставки (радиус 200 м). Если оба совпадения — начислить штраф 500₽ (списание с депозита, транзакция, при необходимости уведомление курьеру и админу).
- Учёт количества отменённых заказов по курьеру (за месяц/всего) и при превышении порога — штраф 1000₽ и/или блокировка.
- Блокировка курьера: уже есть статус BLOCKED в `Courier`; при блокировке по подозрительной активности выставлять этот статус и запрещать брать заказы.

**Отмена и возврат (кто кого уведомляет):**

- Если **магазин отменяет заказ** из своих «📋 Мои заказы» (через текущий флоу `order_cancel_*`):
  - сейчас это возможно только пока заказ в статусе `NEW` (ещё не взят курьером);
  - курьер об этом не уведомляется, т.к. у него ещё нет этого заказа в списке;
  - открытый вопрос к заказчику: нужно ли в будущем уведомлять курьеров о том, что заказ, который они уже видят в «Доступных заказах», был отменён магазином.
- Если **курьер отменяет заказ** (`CANCELLED`) или помечает как **возвращён в магазин** (`RETURNED`):
  - бот уведомляет магазин: кто получатель, адрес и какой курьер отменил/вернул;
  - дальнейшее перераспределение заказа (другому курьеру или отмена окончательно) решает магазин/админ.

**Открытый финвопрос для обсуждения с заказчиком (запомнить и спросить):**

- При отмене:
  - магазином до взятия курьером (`NEW`) — тут штрафов курьеру нет, это чисто проблема магазина;
  - курьером после взятия (`ACCEPTED`/`IN_SHOP`/`ON_WAY`) — как именно делить ответственность:
    - списывать ли фиксированный штраф с курьера;
    - компенсировать ли магазину часть комиссии/стоимости;
    - что считать «нормальной причиной» (клиент отменил, магазин закрылся и т.п.).
- Текущее поведение по деньгам: **никаких автоматических списаний/компенсаций при отмене/возврате ещё нет**, только изменение статуса и уведомления. Фин-модель для отмен нужно будет согласовать с заказчиком отдельно.

**Идея на будущее (фича для обсуждения с заказчиком):**

- Сейчас у курьера есть две отдельные кнопки: «⛔ Отменить заказ» и «↩️ Вернуть в магазин», при этом с точки зрения бизнеса заказ после отказа курьера всё равно не должен «возвращаться в пул», а реальный сценарий один — курьер везёт букет назад в магазин.
- Более логичный UX:
  - сначала у курьера есть только кнопка **«⛔ Отменить заказ»**;
  - после нажатия и подтверждения статус становится чем‑то вроде «отменён курьером, везём обратно», и в `🚚 Мои заказы` вместо «В магазине» / «В пути» появляется одна понятная кнопка **«↩ Вернуть в магазин»** с короткой подписью (чтобы влезла в кнопку);
  - при её нажатии можно:
    - пометить заказ как `RETURNED`,
    - уведомить магазин,
    - опционально дать курьеру ссылку «Построить маршрут обратно в магазин» (Яндекс/2ГИС).
- Это можно продавать как отдельную фичу «нормальный сценарий возврата букета в магазин + маршрут до магазина».
- Оценка по доработке: порядка **6–10 часов разработки** (UX правки кнопок, статусы, уведомления, ссылки на маршрут, тесты), то есть ориентировочно **+10–15k ₽** к бюджету, в зависимости от того, насколько детально делать маршрутизацию и отчётность по возвратам.

## 6. Интеграция с платёжной системой

- **ЮKassa или Stripe:** оплата пополнения депозита.
- Сценарий: курьер нажимает «Пополнить» → ввод суммы → создание платежа в ЮKassa/Stripe → редирект (или deep link) на оплату → webhook при успешной оплате → начисление на баланс + запись в `courier_transactions` (DEPOSIT_TOP_UP).
- Автоматическое списание комиссии и штрафов — только с баланса депозита (не «дергать» карту автоматически без согласия пользователя).

## 7. Остальное по ТЗ для курьеров

- **Возврат заказа в магазин** — новый статус заказа (например RETURNED) или отдельный флоу «вернул в магазин» с причиной; при возврате — проверка гео у магазина при необходимости; комиссию можно не списывать или списать частично (по бизнес-правилам).
- **Отмена заказа** — кто может отменить (курьер/магазин/админ), при отмене вызывать проверку на обход и штрафы.
- **Смена адреса доставки во время выполнения** — разрешать только до определённого статуса (например до «В пути»); пересчёт цены и гео; логирование старого/нового адреса.

## 9. Кратко: что нужно для депозитов

1. **Таблица/поле баланса** — текущий депозит курьера.
2. **Пополнение депозита** — сценарий в боте + платежи (ЮKassa/Stripe) + webhook.
3. **Автоматическое списание комиссии** — при принятии заказа (20% или настраиваемый %), с проверкой достаточности баланса.
4. **Проверка баланса** — вывод в боте (баланс + при необходимости история).
5. **История транзакций** — таблица операций (пополнение, списание комиссии, возврат комиссии, штрафы).

И **геолокация курьера** — обязательный первый шаг, без неё система проверки «был ли курьер в магазине/у получателя» и штрафы не работают.

---

## 10. ЮKassa, самозанятый и денежные потоки

**Модель на первую версию (MVP):**

- Аккаунт ЮKassa регистрируется **на владельца сервиса** (заказчика) — как самозанятый или ИП/ООО.
- В боте используется один `shopId` / `secretKey`, который принадлежит этому аккаунту.
- Курьеры **пополняют депозит** через этот магазин ЮKassa; деньги с платежей сразу падают на счёт заказчика (самозанятого/ИП).
- Бот только создаёт платежи и показывает ссылки/кнопки; карты и платёжные данные хранятся на стороне ЮKassa.

**Как зарабатывает сервис:**

- Доход сервиса закладывается в модель **пополнений депозита курьерами** (комиссия/маржа в тарифах).
- Магазины уже **самостоятельно рассчитываются с курьерами** (вне бота), исходя из своих договорённостей.
- Курьеры при желании могут работать как самозанятые и **выдавать чек магазину** напрямую через «Мой налог» или банковское приложение.

## 11. Магазин → Курьер: прямые выплаты и проверка «оплатил или нет» (черновик)

**Схема без платёжки между ними:**

- В анкете курьера храним его реквизиты для выплат:
  - номер карты / телефон для СБП;
  - при желании — данные самозанятого (ИНН, ФИО) для чеков.
- После того как заказ доставлен и закрыт в боте:
  - магазину показываем сумму к оплате курьеру и его реквизиты;
  - добавляем кнопку **«✅ Оплатил курьеру»**.

**Шаги при нажатии «Оплатил курьеру»:**

1. Магазин нажимает кнопку → бот:
   - ставит внутренний статус оплаты `WAITING_CONFIRM_FROM_COURIER`;
   - отвечает магазину текстом вроде:  
     «Зафиксировал, что ты отметил оплату.  
     Если прикрепишь скрин/фото чека — спор решится быстрее, если вдруг возникнет спор.»
   - даёт возможность отправить фото/скрин чека (опционально).
2. Курьеру уходит запрос:
   - текст: «Магазин X отметил, что оплатил тебе N ₽ за заказ №Y. Всё ок?»;
   - две кнопки:
     - **«✅ Деньги получил»** → статус `PAID_CONFIRMED`;
     - **«❌ Денег не было»** → статус `DISPUTE`.

**Разбор спорных ситуаций:**

- Если курьер нажал **«✅ Деньги получил»**:
  - считаем оплату подтверждённой с двух сторон;
  - заказ помечаем как оплаченный, можно учитывать в отчётах.
- Если курьер нажал **«❌ Денег не было»**:
  - заказ попадает в `DISPUTE`;
  - админам уходит уведомление с пакетом данных:
    - магазин, курьер, сумма;
    - был ли от магазина скрин/чек;
    - время, когда магазин нажал «Оплатил» и когда курьер нажал «Денег не было».
  - дальше решение принимает админ (звонок, запрос доп.скринов, ручное вмешательство).

**Важно:**

- Бот в этой схеме **не видит реальные деньги** и не ходит в банк — это чисто организационный слой:
  - «магазин → отметил оплату»;
  - «курьер → подтвердил/оспорил»;
  - «админ → разбирает спор».
- Реальные переводы (СБП, карта, расчёты самозанятых) остаются на стороне магазинов и курьеров.

**Как это выглядит для налоговой (самозанятый):**

- Самозанятый оказывает **услуги по организации и обеспечению курьерской доставки** (информационный сервис, логистика), а не перепродаёт чужие товары.
- Платёж «пополнение депозита курьера» = предоплата за эту услугу.
- В чеке/описании услуги важно использовать формулировки уровня: «услуги доставки / информационный сервис / логистика», а не «продажа товара».

**Вопрос «магазин платит напрямую курьеру» (на будущее):**

- В первой версии все деньги проходят через аккаунт заказчика (ЮKassa) и дальше распределяются им вручную/через отчёты.
- Схема «магазин → сразу курьеру из бота» требует уже модели маркетплейса и выплат (payouts) и будет рассматриваться как **отдельный этап** с более сложной интеграцией и юридическими договорённостями.

## 12. Будущие фичи (от заказчика)

- **Создание заказа через оператора** — оператор создаёт заявку (ДТП/поломка, указав место курьера).
- **Разовый заказ без регистрации магазина** — магазин создаёт заказ без полной регистрации.
- **Свободный адрес забора** — привезти цветы/клубнику с оптовых магазинов (адрес забора ≠ свой магазин).
- **Филиалы** — один оператор создаёт заказы для сети (напр. ЦВЕТОПТОРГ) с разными адресами.

## 13. Для модуля админов (следующий этап)

**Детали в** `courier-flow-analysis-and-gaps.md`, раздел 13.

- **Задержка возврата комиссии** — при отмене/возврате комиссия не возвращается сразу; админ успевает назначить штраф или подтвердить отмену.
- **Разбор «звоночков»** — уведомления о подозрительных отменах (был в магазине, не у получателя).
- **Назначение штрафов** — админ вручную назначает штрафы (500₽, 1000₽ или произвольную сумму).
- **Время + гео** — запасная верификация при разборе спорных случаев.
