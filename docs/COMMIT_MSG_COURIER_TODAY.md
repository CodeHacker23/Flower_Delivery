## 1. Честное распределение («никого не обижать»)

**Проблема:** Курьер отвёз заказ от магазина на Энтузиастов, вышел снова в линию — ему опять суют ближайший заказ, опять от того же магазина. В итоге он мотается только между этим магазином и зоной; остальные магазины в пролёте — заявки висят, курьер до них не доезжает.

**Что сделали:** Список «Доступные заказы» формируем не просто «по расстоянию», а с честным распределением:

- **Первые 5** — как и раньше, ближайшие по гео (курьеру удобно).
- **Следующие 5** — от магазинов, которым **этот** курьер за последние **24 часа** доставил меньше всего. То есть кому он «должен» — тем и показываем заказы повыше.

Так и «ближайшие» остаются, и ни один магазин не сидит вечно в конце очереди.

### Где трогали код

| Место | Что сделано |
|-------|-------------|
| **OrderRepository** | Метод `findDeliveredByCourierSince(courier, status, since)` — все заказы, которые курьер доставил (DELIVERED) начиная с даты `since`. Нужно для метрики «сколько раз этот курьер отвёз заказы от какого магазина». |
| **OrderService** | `getDeliveredCountPerShopForCourier(courier, lastHours)` — по этим доставкам считаем: по каждому shop_id сколько штук доставил этот курьер за последние N часов. Возвращает `Map<UUID, Long>` (shopId → количество). |
| **OrderService** | `getAvailableOrdersWithFairness(courierLat, courierLon, courier, nearestCount, otherCount)` — список доступных заказов: сначала `nearestCount` штук ближайших по расстоянию, потом до `otherCount` штук от магазинов с минимальным числом доставок этим курьером за 24 ч. Сейчас вызываем с (5, 5). |
| **Bot** | При нажатии «Доступные заказы»: если у курьера есть «свежая» гео (до 30 мин), список строим через `getAvailableOrdersWithFairness(...)` вместо просто «по расстоянию». |
| **CourierAvailableOrdersHandler** | Когда курьер только что отправил гео по запросу «Доступные заказы» — список тоже строим через `getAvailableOrdersWithFairness(lat, lon, courier.getUser(), 5, 5)`. |

Итого: одна и та же логика «честного» списка в двух точках — при уже сохранённой гео (Bot) и при только что полученной гео (CourierAvailableOrdersHandler).

## 2. Проверка гео в радиусе 200 м (включить после тестов)

**Идея:** Подтверждение «В магазине» / «Вручил» считаем действительным только если курьер в радиусе **200 м** от нужной точки:

- для **«В магазине»** — от **магазина** (адрес забора, координаты Shop);
- для **«Вручил»** (один адрес) — от **адреса доставки** заказа (deliveryLatitude/Longitude);
- для **«Вручил» по точке мультиадреса** — от **адреса этой точки** (OrderStop.deliveryLatitude/Longitude).

Одну кнопку «📍 Отправить геолокацию» оставили, вторую «Подтвердить гео» не добавляли. «Второе подтверждение» = как раз эта программная проверка: прислал гео → мы проверили расстояние → ок, тогда меняем статус; не ок — пишем «Вы слишком далеко от точки (более 200 м). Подойдите ближе и отправьте геолокацию снова.» и не меняем статус, ожидание гео не сбрасываем (курьер может подойти и отправить ещё раз).

### Где трогали код

| Место | Что сделано |
|-------|-------------|
| **CourierGeoHandler** | В `handleLocation`: после сохранения lastLocation и до смены статуса — вызов `getReferenceCoordinatesForPending(pending)` (координаты магазина или адреса доставки), затем `GeoUtil.isWithinRadiusKm(lat, lon, refLat, refLon, GeoUtil.RADIUS_200_M_KM)`. Если эталон есть и курьер **вне** радиуса — pending возвращаем в `awaitingLocation`, шлём сообщение «слишком далеко» и кнопку «Отправить геолокацию», выходим. Если эталона нет (магазин/адрес без координат) — проверку не делаем, принимаем гео как раньше. |
| **CourierGeoHandler** | Метод `getReferenceCoordinatesForPending(PendingGeoConfirmation pending)`: для IN_SHOP — заказ с магазином (`getOrderWithShop`), координаты Shop; для DELIVERED без точки — координаты заказа (deliveryLatitude/Longitude); для DELIVERED по точке — заказ + stops, ищем stop по `stopNumber`, координаты этой точки. Возвращает `double[] { refLat, refLon }` или null. |
| **GeoUtil** | Уже были: `RADIUS_200_M_KM = 0.2`, `isWithinRadiusKm(...)` — мы их только используем, не меняли. |

**Важно:** Саму проверку 200 м **закомментировали** для тестов (ты не можешь физически поехать на адрес). В коде остался блок с TODO: «раскомментировать после тестов». Пока он закомментирован — гео принимается без проверки расстояния, как раньше.
